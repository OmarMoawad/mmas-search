# I invented an algorithm to beat binary search. Then I disproved myself.

*How a compiled benchmark and a pile of real data turned my "breakthrough" into an honest lesson about learned indexes — and a working piece of code.*

## The claim

Binary search is the textbook way to find a value in a sorted array: halve the range, compare, repeat, `O(log n)` guaranteed. I thought I could beat it.

My idea — I called it **Multi-Model Adaptive Search (MMAS)** — was this: instead of blindly halving, *look at the data first*. Sample a few points, fit a model to the shape of the values (linear? power-law? exponential?), then **invert the model to jump straight to the answer's index**. If the array is `A[i] = i³` and I want `729`, I don't binary-search — I compute `729^(1/3) = 9` and probe once. One probe, regardless of size. If no model fits, fall back to binary search so I never do worse.

My first benchmark looked spectacular: **1.00 probes per query** on linear, quadratic, cubic, and exponential arrays, versus 9–23 for binary search, across sizes up to 10 million. I wrote it up as a paper. It felt like a breakthrough.

It wasn't. Here's how I found out — and what was actually worth keeping.

## Three ways I was fooling myself

**1. I counted probes, not time.** A binary-search probe is a compare-and-branch — the CPU predicts and pipelines it, nearly free. An MMAS probe additionally computes `pow`, `log`, and a division. Counting "probes" credited my algorithm for work that binary search does *more cheaply per step*. The only honest cost metric in memory is **wall-clock time**, and I had never measured it.

**2. My queries were rigged.** I drew every query *from the array*, and my arrays were exact formulas (`i`, `i²`, `i³`, `2^i`). Inverting the fitted cubic to find a value I *know* is on the cubic isn't a search result — it's a property of the setup. Worse, my "linear" case was `A[i] = i`, so the winning trick was literally `A[v] == v`: the value equals its own index. That's not searching.

**3. I never touched real data.** Every "win" was on an array generated by the exact formula I then inverted — which is precisely the case where you wouldn't need a search structure at all. If your data *is* `i³`, you can compute any index directly; you don't store it and search.

## The honest rebuild

So I rebuilt the evaluation to remove every excuse:

- **Compiled C++** (MSVC `/O2`), all algorithms in one language, timed in nanoseconds.
- **`std::lower_bound`** — the industry-optimized binary search — as the baseline to beat.
- **Mixed queries: 50% present, 50% absent**, every result verified against `std::binary_search` (zero errors).
- **Real data** harvested from my own filesystem: ~81k unique file sizes (heavy-tailed) and ~9.5k unique file-modification timestamps (clustered, no clean formula).

And I added the algorithm the field actually uses for messy data: a **piecewise-linear index** (`pgm_pla`) that approximates the key→position function with a chain of short line segments, each guaranteed accurate to within ±ε — so it needs *no* global formula.

## What the numbers actually said

| dataset | binary (ns) | interpolation (ns) | mmas_closed (ns) | **pgm_pla (ns)** |
|---|--:|--:|--:|--:|
| real file sizes | 72 | **377,294** | 75 | **62** |
| adversarial | 68 | 109,242 | 74 | **32** |
| exponential | 140 | 1,333,832 | 138 | **84** |
| uniform | 138 | 87 | 101 | **86** |

Three lessons fell out:

**Plain interpolation search is dangerous.** On uniform data it's brilliant. On real, non-uniformly-spaced data it's catastrophic — up to **~9,000× slower** than binary search, because its position estimate assumes even spacing that real data doesn't have. Never deploy it uncapped.

**My "1 probe" magic only existed on the synthetic formulas.** On *every* real dataset, MMAS's model detector correctly reported "no closed-form fit" and fell back to binary search — matching it exactly. Safe, but no gain. Exactly as it should be, and exactly the opposite of a breakthrough.

**The piecewise index was the real win.** With ~11 array probes versus binary's ~16, it suffers fewer cache misses and actually *beat* binary search on wall-clock — even in RAM — on the larger real dataset and on adversarial data. And its lead widens dramatically once each probe is expensive.

## The part that changes the verdict: expensive probes

Here's the nuance I'd missed. Probe count is the *wrong* metric when data is in fast RAM — but the *right* metric when each probe is a **cache miss (~100 ns), an SSD read (~50 µs), or a network round-trip (~1 ms)**. In that regime, latency ≈ probes × cost-per-probe, and cutting 16 probes to 11 is real, compounding time.

That's not a hypothetical — it's the entire reason **learned indexes** exist in real databases: on huge, on-disk, read-mostly data, fewer probes means less I/O. My piecewise variant lands squarely in that niche.

## What I actually built (and what I learned)

The honest scorecard:

- **Binary search** is the robust default: never catastrophic, fastest on small arrays.
- **Interpolation search** is a trap on real data.
- **Closed-form MMAS** only helps on data following an exact analytic law — which you could address directly anyway.
- **The piecewise index** is genuinely useful, and it's the mechanism behind the **PGM-index** (Ferragina & Vinciguerra, VLDB 2020) and learned indexes generally (Kraska et al., SIGMOD 2018).

In other words, my "invention" was a from-scratch rediscovery of a real, published idea — and the most valuable thing I produced wasn't the algorithm. It was the discipline to **build the experiment that could prove me wrong, run it, and believe the result.**

The code, the compiled benchmark, the real-data harness, the piecewise index wired into the search dispatcher with passing tests, and the honestly-rewritten paper are all here: **[github.com/OmarMoawad/mmas-search](https://github.com/OmarMoawad/mmas-search)**.

If there's one thing I'd want a reader (or a future employer) to take from it: an idea that only wins on the benchmark you designed to make it win isn't a result yet. Go find the data that doesn't care about your idea, and measure the thing that actually costs money. Sometimes it survives. Sometimes it teaches you what was real all along.
